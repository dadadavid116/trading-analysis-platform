================================================================================
  PART 1 — WHAT THIS PLATFORM IS BECOMING
================================================================================

This platform is no longer a "crypto dashboard with more panels."

It is a CRYPTO SELF-TRADING OPERATING SYSTEM built in layers.

The mental model is a hybrid terminal + dashboard — similar in shape to
professional trading operating environments, but crypto-native in its
data sources, context, and execution logic.

─── THE FOUR WORKSPACES ────────────────────────────────────────────────────────

  WORKSPACE 1: MARKET INTELLIGENCE
  ─────────────────────────────────
  Live market structure, AI interpretation, alerting, depth/liquidation
  analysis, derivatives context, funding/OI/long-short ratio monitoring.
  This is where the trader understands the environment.

  WORKSPACE 2: OPERATOR CONSOLE
  ─────────────────────────────
  Scanner runs, candidate lists, filter outputs, score breakdowns, skip
  reasons, terminal-style event logs, signal queue.
  This is where the trader finds and evaluates setups.

  WORKSPACE 3: EXECUTION & RISK
  ─────────────────────────────
  Position proposals, order states, paper execution flow, risk engine,
  open risk exposure, kill switch.
  This is where the trader acts — controlled, rules-based, AI-assisted.

  WORKSPACE 4: REVIEW & IMPROVEMENT
  ──────────────────────────────────
  Backtesting, replay, journaling, performance analytics, strategy
  versioning, post-trade review, AI score tracking.
  This is where the trader learns and improves.

─── CRYPTO-NATIVE, NOT STOCK-TRANSLATED ────────────────────────────────────────

The reference for this design comes partly from professional stock-platform
terminal UX. The correct translation approach is:

  KEEP from reference:
    - terminal + dashboard hybrid shape
    - scanner / signal / risk / execution / review progression
    - structured operator workflow (not just a single view)
    - account tracking, position state, logs, scores

  REPLACE with crypto-native equivalents:
    - premarket gap context  →  funding rate + OI + liquidation context
    - earnings catalyst      →  on-chain events, macro news, funding spikes
    - share volume           →  perpetual contract open interest delta
    - level 2 quotes         →  Binance/OKX order book depth + imbalance
    - relative strength vs sector  →  multi-asset relative performance (BTC/ETH/SOL)

─── PRIMARY DATA SOURCE CHANGE ─────────────────────────────────────────────────

The platform's primary price and contract data source should switch from:
  Binance BTCUSDT Spot
to:
  OKX BTCUSDT Perpetual Swap Contract

This is a configuration/implementation adjustment, not an open roadmap question.
All new collector, analysis, and risk logic should be built against the perpetual
contract unless a specific spot context is required.

These questions are answered. They should not be reopened or treated as
unresolved in future phase planning.

     - command menu (BotFather menu configuration)

   The current overlap between web chart analysis and Telegram /analysis is
   a product naming issue, not a technical problem.

     - one command/path = market commentary (broader context, narrative)

   redesigns the analysis panel (Phase 26).

   Switch to OKX BTCUSDT perpetual swap as the primary data source.
   This affects: price collectors, chart feeds, analysis context, alert thresholds.
   Implementation detail for Phase 27/28, not an ongoing debate.

   Questions still open:

================================================================================
  PART 4 — ANALYSIS FEATURE REFRAME
================================================================================

The current chart analysis feature is a MARKET INTERPRETATION TOOL.
It answers: "What does the chart look like?"

It must evolve into a TRADE DECISION ANALYSIS ENGINE.
It must answer: "Given this trader's style, risk tolerance, holding period,
target return, and strategy source — does this setup produce a valid trade plan?"

─── NEW INPUTS REQUIRED ────────────────────────────────────────────────────────

  These inputs should eventually be collected from a trader profile and fed
  into the analysis engine before generating a result:

    - Trader profile / trading style (momentum, mean-reversion, breakout, etc.)
    - Risk per trade (as % of account or fixed amount)
    - Preferred holding duration (scalp / intraday / swing / position)
    - Target R multiple or return profile (e.g. 2R, 3R minimum)
    - Strategy source:
        imported strategy (from backtested rules)
        user-written idea (free-text direction)
        platform-generated setup (from signal engine)

─── NEW OUTPUT BEHAVIOR REQUIRED ────────────────────────────────────────────────

  The analysis engine should eventually produce:

    - A trade plan, not just market commentary
    - Invalidation logic (the condition that cancels the setup)
    - Setup expiration (time window in which the setup remains valid)
    - Manual cancellation support (user can clear a setup)
    - Annotation cleanup on chart when setup expires / is invalidated / superseded:
        expired
        invalidated
        user cleared
        superseded by new analysis

─── CURRENT STATE ───────────────────────────────────────────────────────────────

  Phase 23/24b introduced direction-aware analysis with Auto/Long/Short bias.
  This was the right first step.
  The full reframe (trader profile inputs, trade plan output, expiry logic)
  belongs in Phase 26 (Market Intelligence Center) and matures through
  Phase 30 (Signal Engine) and Phase 31 (Risk Engine).


================================================================================
  PART 5 — NEW STRUCTURAL ADDITIONS
================================================================================

A. ACCOUNT WORKSPACE
   This is a new top-level platform section — not a small widget.
   It is a separate interface that tracks trader performance over time.

   Sections:
     Account Total
       - Balance / equity curve chart
       - Return (daily / total)
       - Net Profit
       - Daily Gain / Loss

     Trades
       - Win / loss ratio (day / month / year)
       - Total profit / loss (day / month / year)

     AI Trading Score
       - Scored against user-defined goals
       - Tracks direction consistency
       - Tracks execution quality
       - Tracks adherence to strategy and risk rules

   This absorbs Phase 32 (Account State Layer) in the new roadmap.

B. BACKTESTING AS CORE MODULE
   Backtesting is not a side feature. It is part of the self-trading discipline loop.
   It should include:
     - Strategy replay on historical data
     - Setup validation (was the entry valid given the rules?)
     - Outcome analysis (how did the setup resolve?)
     - Performance analytics (win rate, drawdown, R-multiple distribution)
     - Parameter comparison (which settings produce better results?)
     - Visualized testing interface (chart replay with annotations)
   This is Phase 34 in the new roadmap.

================================================================================
  PART 7 — FUTURE PHASES (25 THROUGH 40)
================================================================================

These phases are ordered by priority. Infrastructure and architecture come
before features. Operator workflow comes before UI polish. Execution comes
after risk. Live trading is gated last.

───────────────────────────────────────────────────────────────────────────────
PHASE 25 — DATA FOUNDATION, GOVERNANCE, AND OBSERVABILITY
Type: Structural refactor
Priority: Highest — do this before any major new feature work
───────────────────────────────────────────────────────────────────────────────

  Why now:
  The platform is live and useful, but schema changes are fragile (ALTER TABLE
  on startup), there are no backups, and failures are invisible unless you check
  logs manually. This must be solved before the system grows more complex.

  Deliverables:
    - Alembic migrations: replace startup ALTER TABLE approach with versioned
      migration files. Enables schema rollback and reproducible deployments.
    - Backup policy: automated daily pg_dump to a named volume or external
      storage. Retention: 7 daily, 4 weekly. Telegram alert if backup fails.
    - Service health panel: a page (or header indicator) showing per-service
      last heartbeat, data freshness, and connection state for all collectors.
    - Unified event log: structured logs for collector connects/disconnects,
      analysis worker runs, alert evaluations, errors, Telegram commands.
    - Operational alerts: internal alerts that fire when collectors drop,
      DB falls behind, or worker stops emitting events.

  Absorbs from old roadmap:
    - Alembic migrations (old Phase 42)
    - Automated DB backups (old Phase 43)

───────────────────────────────────────────────────────────────────────────────
PHASE 26 — MARKET INTELLIGENCE CENTER
Type: Major product upgrade
Priority: High — first visible product-level upgrade after foundation
───────────────────────────────────────────────────────────────────────────────

  Why now:
  The current Analysis panel is minimal. The chart analysis feature exists but
  is still fundamentally a market commentary tool. This phase redesigns both
  and begins the shift toward a trade decision analysis engine.

  Deliverables:
    - Redesigned Analysis panel: rich display of scheduled summaries with
      timestamp, model badge, market snapshot at generation time, expandable
      sections, copy button, history of last 5-10 summaries.
    - Event timeline: a live timeline of significant market events (alert
      triggers, analysis runs, large liquidations, analysis results).
    - Depth visualization: order book depth chart showing cumulative bid/ask
      volume by price level, liquidity walls, bid/ask imbalance ratio.
    - Liquidation aggregation: rolling window stats (how many liquidations in
      last 5m/15m/1H, total USD value, side breakdown).
    - Analysis result lifecycle:
        - Setup expiration (user-set or time-based)
        - Invalidation trigger (price crosses stop without entry)
        - Manual cancellation (user clears the current analysis)
        - Superseded state (new analysis replaces old one)
        - Chart annotation cleanup on any of the above states
    - Naming clarity: separate "market commentary" from "trade setup analysis"
      at the UI and API level.
    - Begin collecting trader profile inputs (trading style, risk per trade,
      holding preference) as optional context for the analysis engine.

  Absorbs from old roadmap:
    - Richer Analysis Panel (old Phase 28)
    - Order Book Depth Chart (old Phase 29)

───────────────────────────────────────────────────────────────────────────────
PHASE 27 — CRYPTO CONTEXT COLLECTOR PACK
Type: Structural + data expansion
Priority: High — prerequisite for any meaningful signal or scoring logic
───────────────────────────────────────────────────────────────────────────────

  Why now:
  The platform currently collects price, liquidations, and order book data.
  That is necessary but not sufficient for crypto-native market intelligence.
  Without derivatives context, the analysis engine is working with incomplete
  information. This is the crypto-native replacement for "premarket gap context"
  in stock terminal workflows.

  Deliverables:
    - Funding rate collector: poll OKX/Binance funding rate endpoint on 8H
      cycle + current estimated rate. Store in funding_rates table.
    - Mark price and premium collector: mark price vs last price vs index price.
      Store delta (basis). Surface basis in market intelligence view.
    - Open interest collector: current OI, OI delta (1H/4H/24H). Store in
      open_interest table. Surface as chart overlay option.
    - Long/short ratio collector: top trader long/short ratio, global account
      long/short ratio. Store in ls_ratios table.
    - Market context snapshots: at each analysis run, capture a structured
      snapshot of all derivatives context and attach it to the analysis record.
    - API endpoints: expose all new data via /api/derivatives/* endpoints.
    - Frontend: minimal display widgets for funding rate, OI, and basis.
      Full visualization comes in Phase 26 / later.

  Absorbs from old roadmap:
    - Funding Rate Monitor (old Phase 34)
    - Open Interest Tracker (old Phase 35)

───────────────────────────────────────────────────────────────────────────────
PHASE 28 — MULTI-ASSET UNIVERSE + EXCHANGE ALIGNMENT
Type: Structural platform expansion
Priority: High — enables scanner and signal logic to operate across assets
───────────────────────────────────────────────────────────────────────────────

  Why now:
  A signal engine cannot operate on one asset. A scanner is meaningless with
  one symbol. This phase turns the platform from a single-asset monitor into
  a real multi-asset operator base.

  Deliverables:
    - Primary source switch: move all BTCUSDT data collection to
      OKX BTCUSDT perpetual swap contract as the primary feed.
      Retain Binance as a secondary/fallback source.
    - Symbol registry: a configuration-driven asset universe table.
      Initial set: BTCUSDT, ETHUSDT, SOLUSDT perpetuals.
      Extensible to additional assets without code changes.
    - Multi-symbol routing: collectors, analysis workers, and alert evaluators
      all become symbol-aware. Each symbol runs independently.
    - Symbol selector: UI element in the header or PricePanel to switch the
      active symbol on all panels simultaneously.
    - Relative strength indicator: simple multi-asset comparison — percentage
      change over 1H/4H/24H across all tracked assets.
    - All DB tables already have symbol column — routing update only.

  Absorbs from old roadmap:
    - Multi-Asset Support (old Phase 31)
    - Multi-Exchange Data (old Phase 44, partially)

PHASE 29 — OPERATOR CONSOLE / TERMINAL MODE
Type: Major workflow upgrade (new workspace: Operator Console)
Priority: High — the central product differentiator
───────────────────────────────────────────────────────────────────────────────

  Why now:
  This is the feature that turns the platform into a professional operating

PHASE 30 — SIGNAL ENGINE V1
Type: Major logic layer (populates Operator Console output)
Priority: High — first time the platform generates its own structured signals
───────────────────────────────────────────────────────────────────────────────

  Deliverables:
    - Signal object schema: structured record with fields:
        symbol, timeframe, direction (long/short), setup_type,
        confidence_score, entry_zone, stop_loss, take_profit[],
        invalidation_condition, expiry_time, context_snapshot,
        ai_explanation, strategy_source, status
    - L1 / L2 filter logic:
        L1 = broad universe filter (volatility, volume, trend alignment)
        L2 = precise setup filter (structure, key levels, derivatives context)
    - Setup types: initial set of named setup types to recognize
        (e.g. breakout retest, range compression, OI expansion with trend)
    - Trader profile integration: signal generation reads trader style,
      holding preference, and R target before scoring.
    - AI explanation: Claude writes a 2-3 sentence rationale for each
      signal and stores it with the signal record.
    - Signal status lifecycle: pending → active → expired / invalidated / hit

  Note: Signal engine produces candidates for the operator to review.
  It does not automatically execute. Human review + approval is always required.

───────────────────────────────────────────────────────────────────────────────
PHASE 31 — RISK ENGINE V1
Type: Major logic layer
Priority: High — required before any execution or position management
───────────────────────────────────────────────────────────────────────────────

  Deliverables:
    - Per-trade risk calculator:
        input: account size, risk %, entry, stop → output: position size
        support for both fixed $ risk and % risk modes
    - Portfolio-level open risk cap:
        maximum total open risk across all positions (e.g. 6% of account)
        new positions blocked if cap would be exceeded
    - Stop logic types:
        structure-based (below/above key level)
        ATR-based (N × ATR from entry)
        user-defined (manual price input)
    - Scale-in / add-on rules: define conditions under which a position
      can be added to (price confirmation, new structure confirmation)
    - Risk events: log all risk evaluations, limit breaches, size overrides
    - Kill switch: single action that cancels all pending signals and closes
      or flags all open positions for immediate review
    - AI trading score foundations: start tracking execution adherence to
      risk rules as a scored metric over time

PHASE 33 — PAPER EXECUTION ADAPTER
Type: Execution workflow layer
Priority: Medium-high — first time the platform acts, not just observes
───────────────────────────────────────────────────────────────────────────────

  Deliverables:
    - Paper-only execution: all order actions simulate fills at current price.

        view open positions
        confirm a pending proposal

PHASE 34 — REPLAY AND BACKTESTING
Type: Review / research layer (new workspace: Review & Improvement)
Priority: Medium-high — required for strategy validation before live gate
───────────────────────────────────────────────────────────────────────────────

  Deliverables:
    - Historical chart replay: step through historical candles bar by bar,

    - Performance metrics: win rate, max drawdown, Sharpe ratio,
      average R-multiple, profit factor, total trades
    - Parameter comparison: test two strategy configurations side by side
    - Strategy version management: save and compare multiple strategy versions

Priority: Medium — future-proofs without blocking current work
───────────────────────────────────────────────────────────────────────────────

PHASE 40 — LIVE EXECUTION GATE
Type: Controlled long-term gate — final phase before real capital
Priority: Lowest — do not rush; all prior phases must be stable first
───────────────────────────────────────────────────────────────────────────────

  PREREQUISITES — ALL must be met before live execution:
    ✓ Paper execution is stable and has been used through Phase 33
    ✓ Risk engine is proven and has not been bypassed (Phase 31)
    ✓ Backtesting / replay is complete and strategies are validated (Phase 34)
    ✓ Review loop is in active use (Phase 36)
    ✓ Deployment is stable (Phase 25 + ongoing monitoring)
    ✓ VPS is on Tokyo or Singapore node for < 10ms latency to OKX
    ✓ A separate test environment exists for live-order testing (not production)

  Deliverables:
    - Exchange API connection (OKX, trading key, withdraw DISABLED)
    - Staged rollout: paper → micro live (very small size) → full live
    - Strict human approval flow: every order proposal must be explicitly
      confirmed before submission. No autonomous order placement.
    - Emergency controls:
        Cancel all open orders (single action)
        Flatten all positions (single action)
        Disconnect exchange (removes API key from active session)
    - Permission isolation: live trading permissions are separate from
      analysis / alert permissions. Require explicit enablement.
    - Production/live separation: live trading runs in an isolated container
      with its own credentials. Never shares state with dev/test environment.
    - Risk-linked execution guardrails: position size, open risk cap, and
      daily loss limit are enforced in the execution adapter before any order
      is submitted to the exchange.

  WARNING: No real capital should be at risk until every prerequisite above
  is met and has been stable for at minimum 30 days in paper mode.

================================================================================
  PART 9 — LOCAL AI / LOCAL DEPLOYMENT DIRECTION
================================================================================

Position: This is a future architecture track, not an immediate phase.

Do not block current product evolution on local deployment questions.

The platform should remain internet-connected for all real-time market data.
Live data feeds (OKX WebSocket, news APIs, derivatives context) require
internet access and cannot be meaningfully replicated locally.

Suggested framing for future architectural decision:

  Layers that should STAY API-based (reasoning + context required):
    - Chat / interpretation layer (Claude / GPT-4o)
    - Strategy explanation and rationale generation
    - News summarization with live context
    - Trade decision analysis engine

  Layers that COULD move to local inference (when capable models exist):
    - Pattern recognition / scoring (deterministic rules + local model)
    - News sentiment classification (fine-tuned small model)
    - Alert condition evaluation (already deterministic — stays rules-based)
    - Journal entry analysis (low-latency batch task, no real-time need)

  Layers that should NEVER be AI-dependent (must be deterministic):
    - Risk engine enforcement (position sizing, loss limits, kill switch)
    - Order submission guardrails
    - Authentication and access control

  Recommended timeline:
    - Phase 25-36: stay fully API-based, optimize for capability
    - Phase 37+: evaluate local model candidates based on maturity
    - Post-Phase-38: consider hybrid architecture (local fast path + API fallback)
