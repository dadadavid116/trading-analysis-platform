================================================================================
  SECTION 1 — WHAT THE PLATFORM IS
================================================================================

A personal, 24/7 crypto market monitoring and trading analysis platform focused
on BTC/USDT. It runs entirely in Docker Compose on a Linux VPS. The platform
combines live market data collection, AI-powered analysis, alert management,
and a Telegram bot into one unified system.

The web dashboard is protected by HTTP Basic Auth (Caddy layer). There is no
public-facing login page — access is restricted by credentials in the .env file.

Core capabilities today:
  - Live BTC/USDT candlestick chart (powered by lightweight-charts + Binance API)
  - Real-time price stats panel (SSE stream from backend database)
  - Liquidations panel (Binance Futures forceOrder stream)
  - Order book panel (Binance depth20 stream)
  - Alerts panel — create, monitor, and delete price/liquidation alerts
  - Desktop browser notifications when alerts trigger
  - AI Chat panel — Claude Sonnet or ChatGPT, with full tool use (alerts, price)
  - Chart Analysis feature — Claude reads last 50 candles, draws support/resistance/
    entry/stop loss/take profit lines on the chart, pushes formatted analysis to chat
  - Strategy Validator — OpenAI validates strategy, Claude writes summary and sets alerts
  - Telegram bot — full parallel interface to the web dashboard
  - Chat history — sessions persisted to database, exportable to .md files
  - HTTPS with auto-renewing Let's Encrypt certificate via Caddy

FRONTEND

  DB driver:        asyncpg (PostgreSQL async driver)
  Validation:       Pydantic v2 (request/response schemas)
  Config:           pydantic-settings (env vars → typed Settings object)
  WebSocket client: websockets library (Binance streams)
  HTTP client:      httpx (async, used for Binance REST and AI APIs)
  AI:               anthropic SDK (Claude), openai SDK (GPT-4o)
  Telegram:         python-telegram-bot v21

DATABASE
  Engine:     PostgreSQL 16 (Docker container)
  Tables:     price_candles, liquidations, orderbook_snapshots,
              alerts, analysis_summaries, chat_sessions, chat_messages
  Schema mgmt: SQLAlchemy create_all on startup + inline ALTER TABLE migrations
               (no Alembic yet — planned for future)

DEPLOYMENT
  Hosting:       Hetzner CX22 VPS (2 vCPU, 4 GB RAM, 40 GB SSD, ~$4.50/month)
  OS:            Ubuntu 24.04
  Containers:    Docker + Docker Compose
  Reverse proxy: Caddy 2 (HTTPS termination, Basic Auth, reverse proxy)
  Domain:        DuckDNS subdomain (free dynamic DNS)
  TLS:           Let's Encrypt via Caddy ACME (auto-renewing)
  Services:      9 Docker Compose services (see Section 4)

  SECTION 4 — DOCKER COMPOSE SERVICES (PRODUCTION)

Service        Image              Role
─────────────────────────────────────────────────────────────────────────────
db             postgres:16        PostgreSQL database. Internal only. Persistent
                                  named volume. Runs init_db.sql on first start.

api            ./backend          FastAPI + Uvicorn on port 8000. Serves all
                                  /api/* endpoints. Internal only (Caddy proxies).

frontend       ./frontend         Multi-stage: Vite build → Nginx on port 80.
                                  Serves static React bundle. Internal only.

collector      ./backend          Runs collectors/run_all.py — connects to three
                                  Binance WebSocket streams simultaneously:
                                    • btcusdt@kline_1m → price_candles table
                                    • btcusdt@forceOrder → liquidations table
                                    • btcusdt@depth20 → orderbook_snapshots table
                                  Each collector has its own reconnect loop.

analysis       ./backend          Runs analysis/run.py — scheduled worker that
                                  calls Claude API every N minutes (configurable),
                                  generates a BTC market summary, stores in DB.

alerts         ./backend          Runs alerts/run.py — polls DB every 1 minute,
                                  evaluates all active alert rules against latest
                                  price/liquidation data, fires Telegram notification
                                  when a threshold is crossed.

telegram       ./backend          Runs telegram_bot/run.py — long-polling Telegram
                                  bot. No inbound ports needed. All actions sync
                                  through the shared database.

chat_export    ./backend          Runs chat_export/run.py — exports chat sessions
                                  from DB to .md files in /app/chat_history/ volume.

caddy          caddy:2-alpine     Single public-facing service. Ports 80 + 443.
                                  Handles HTTPS (Let's Encrypt), HTTP Basic Auth,
                                  and routes:
                                    /health     → api:8000 (no auth)
                                    /api/*      → api:8000 (with SSE flush_interval)
                                    /*          → frontend:80
                                  Env vars: DOMAIN, CADDY_USER, CADDY_HASHED_PASSWORD

================================================================================
  SECTION 6 — DATABASE SCHEMA
================================================================================

price_candles
  id          SERIAL PRIMARY KEY
  symbol      VARCHAR(20)         e.g. "BTCUSDT"
  timestamp   TIMESTAMP WITH TZ   candle close time (1-minute candles)
  open        NUMERIC
  high        NUMERIC
  low         NUMERIC
  close       NUMERIC
  volume      NUMERIC
  UNIQUE INDEX on (symbol, timestamp) — enables upsert on every Binance tick

liquidations
  id          SERIAL PRIMARY KEY
  symbol      VARCHAR(20)
  timestamp   TIMESTAMP WITH TZ
  side        VARCHAR(10)         "buy" (long liquidated) or "sell" (short liquidated)
  price       NUMERIC
  quantity    NUMERIC
  exchange    VARCHAR(50)         "binance"

orderbook_snapshots
  id          SERIAL PRIMARY KEY
  symbol      VARCHAR(20)
  timestamp   TIMESTAMP WITH TZ
  bids        JSONB               Array of [price, quantity] pairs, descending
  asks        JSONB               Array of [price, quantity] pairs, ascending

alerts
  id              SERIAL PRIMARY KEY
  name            VARCHAR
  symbol          VARCHAR(20)
  condition_type  VARCHAR(50)     "price_above", "price_below", "liquidation_spike"
  threshold       NUMERIC
  window_minutes  INTEGER NULLABLE  (for liquidation_spike only)
  trigger_mode    VARCHAR(10)     "once" or "rearm"
  is_active       BOOLEAN
  triggered_at    TIMESTAMP NULLABLE
  created_at      TIMESTAMP WITH TZ

analysis_summaries
  id            SERIAL PRIMARY KEY
  symbol        VARCHAR(20)
  generated_at  TIMESTAMP WITH TZ
  summary_text  TEXT
  model_used    VARCHAR(50)

chat_sessions
  id              SERIAL PRIMARY KEY
  platform        VARCHAR(20)     "web" or "telegram"
  model           VARCHAR(50)     "claude" or "openai"
  title           VARCHAR NULLABLE  (first user message, trimmed)
  created_at      TIMESTAMP WITH TZ
  last_active_at  TIMESTAMP WITH TZ

chat_messages
  id          SERIAL PRIMARY KEY
  session_id  INTEGER FK → chat_sessions
  role        VARCHAR(20)    "user" or "assistant"
  content     TEXT
  created_at  TIMESTAMP WITH TZ

================================================================================
  SECTION 8 — AI FEATURES IN DETAIL
================================================================================

1. WEB CHAT PANEL (Claude or ChatGPT)
   Model: claude-sonnet-4-6 or gpt-4o
   Tools available: get_current_price, create_price_alert, list_alerts, delete_alert
   System prompt includes: live BTC price snapshot + timeframe reminder
   Session persistence: every web conversation stored in chat_sessions / chat_messages
   History: full conversation history sent on each request (up to 12 turns)

2. CHART ANALYSIS (Claude)
   Triggered by: "✦ Analyze" button in PricePanel
   Inputs: last 50 BTC klines fetched from Binance (via backend service), timeframe,
           optional user_bias ("bullish — looking for a long setup" / "bearish...")
   Prompt logic:
     - If BULLISH: long setup — entry near support, SL below support, TP at resistance
     - If BEARISH: short setup — entry near resistance, SL above resistance, TP at support
     - If SIDEWAYS: Claude picks more probable direction from momentum
   Output JSON: trend, direction (long/short), support_levels[2], resistance_levels[2],
                entry_zone {low, high}, stop_loss, take_profit[2], reasoning, timeframe
   Frontend actions:
     - Draws colored price lines on chart (green=support, red=resistance, blue=entry,
       orange=stop loss, red=TP)
     - Formats markdown analysis message and injects it into ChatPanel
     - ChatPanel automatically opens if it was collapsed

3. STRATEGY VALIDATOR (OpenAI → Claude pipeline)
   Triggered by: "Strategy" button in ChatPanel input row
   Stage 1: OpenAI (gpt-4o) receives strategy description, returns JSON:
            { valid, reason, name, entry_condition, exit_condition, timeframe,
              stop_loss, take_profit }
   Stage 2: If valid, Claude writes a 2-3 sentence plain-English summary
   Frontend: Shows StrategyCard with all fields + "Approve & Set Alert" button
   On approve: sends structured approval message to Claude with tool use to create alerts

4. SCHEDULED ANALYSIS WORKER
   Container: analysis
   Interval: ANALYSIS_INTERVAL_MINUTES (env var, default ~10 min)
   Process: Claude reads latest price + liquidations from DB, writes summary to
            analysis_summaries table
   API access: GET /api/analysis/latest (shown in AnalysisPanel — currently basic)

5. TELEGRAM BOT
   Commands:
     /start              Welcome message
     /help               Full command reference
     /price              Live BTC price (same data as PricePanel)
     /status             Platform health, data freshness, alert count, AI model
     /alerts             List all active alerts with IDs
     /setalert above/below <price>  Create price alert directly
     /delete_alert <id>  Delete alert (appears removed on web chart within 15s)
     /analysis           On-demand market analysis (Claude or ChatGPT)
     /strategy <text>    Strategy validator with approve/dismiss inline buttons
     /model              Show or switch AI model
     /claude             Switch to Claude Sonnet
     /chatgpt            Switch to ChatGPT (GPT-4o)
     /clear              Clear conversation history
   Free text: routed to selected AI model with tool use (price, alerts)
   Sync: all alert changes are visible on the web dashboard within 15 seconds
   Access control: all commands restricted to TELEGRAM_CHAT_ID
   Session persistence: Telegram conversations stored in same DB as web sessions


================================================================================
  SECTION 9 — ALERT SYSTEM IN DETAIL
================================================================================

Condition types supported:
  price_above          Triggers when latest 1m candle close > threshold
  price_below          Triggers when latest 1m candle close < threshold
  liquidation_spike    Triggers when liquidation event count in last window_minutes > threshold

Trigger modes:
  once    Alert triggers once, stays in "triggered" state until deleted
  rearm   Alert resets automatically when condition is no longer met, can trigger again

Evaluation cycle: alerts worker polls every ALERT_EVALUATION_INTERVAL_MINUTES (default 1)
Notification channels:
  - Telegram message (if TELEGRAM_BOT_TOKEN + TELEGRAM_CHAT_ID configured)
  - Browser desktop notification (if user has granted permission in browser)

Alert lifecycle:
  1. Created via web form, click-on-chart popover, ChatPanel AI, Telegram /setalert,
     or Telegram AI free text
  2. Alert line drawn on PricePanel chart within 15 seconds (dashed orange)
  3. Alert worker evaluates on next 1-minute cycle
  4. When triggered: Telegram notification sent, triggered_at set in DB,
     browser desktop notification fires on next 15s frontend poll
  5. Deleted via: web × button, ChatPanel AI (delete_alert tool), /delete_alert Telegram command

================================================================================
  SECTION 11 — CURRENT PLATFORM LIMITATIONS / KNOWN GAPS
================================================================================

1. Analysis panel (AnalysisPanel.tsx) — exists but is minimal. The scheduled worker
   generates summaries but the panel does not display them richly. Needs redesign.

2. Single asset only — entire platform is hardcoded to BTC/USDT. No multi-asset
   support. Symbol switching is not implemented anywhere.

3. No Alembic migrations — schema changes are handled via inline ALTER TABLE at
   startup. Will become fragile as schema grows.

4. No backtesting — alerts and analysis are forward-looking only. No historical
   strategy evaluation.

5. Mobile layout — dashboard is desktop-only. No responsive breakpoints.

6. No multi-user support — single credentials via Caddy Basic Auth.

7. OpenAI API key — ChatGPT and strategy validation are optional features that
   require OPENAI_API_KEY to be set. Currently returning 429 errors if free tier.

8. Order book display — shows top 5 bids/asks but no depth visualization (no bar
   chart, no cumulative depth chart).

9. Alert rearm logic — "rearm" mode works but there is no visual history of how
   many times an alert has fired.

10. Telegram bot — long-polling mode only. No webhook mode. Bot conversation
    history is stored in memory per process restart (chat_data). Sessions persist
    to DB but bot restart loses the in-memory turn count.
